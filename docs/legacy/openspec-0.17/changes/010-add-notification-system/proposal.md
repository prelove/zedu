# Proposal: 通知与提醒系统

## Why
"上课提醒容易遗漏、续费跟进被动"是Zedu项目最初的立项动机（见PRD第一章
1.3节背景描述）。本change是把这个核心价值真正兑现的地方——如果这里做得
不可靠，整个系统相对Excel+人工的优势就不成立了。

六类定时任务中，课前提醒和自动关闭过期课次要求严格的幂等性（任务可能被
重复触发，不能重复发邮件或误关正在进行的课次），通知失败要有可靠的重试
机制而不是发出去就不管了。

## 业务背景
根据PRD第二十一章21.2节的六类定时任务设计，每一类任务对应一个真实的
运营痛点：
- 课前提醒（任务1）：直接对应"上课忘记提醒"这个最原始的立项动机
- 教务晨报（任务2）：对应"每天要花时间翻Excel看今天有什么事"这个
  日常摩擦，把这个摩擦变成一封自动发送的邮件
- 余额预警（任务3）：对应"续费跟进被动"，学生余额见底了运营者却
  没注意到，导致学生上完课发现没法继续
- 老板周报（任务4）：面向经营决策，让老板不需要打开系统也能定期
  了解经营状况
- 通知重试（任务5）：邮件服务本身可能因为网络、Resend服务临时不可用
  等原因失败，这是外部依赖的正常波动，不能因为一次失败就永久放弃
- 自动关闭过期课次（任务6）：防止Operator忘记做课后确认导致课次
  永远挂起在"待处理"状态，工作台的"待确认"提示会持续累积失去参考价值

## What Changes
- 新增NotificationSender接口，实现ResendSender(主)和SmtpSender(备用降级)
- 新增5个notification_template种子数据（课前提醒×2、余额预警、晨报、周报）
- 新增六类定时任务：课前提醒/教务晨报/余额预警/老板周报/通知重试/
  自动关闭过期课次
- 新增通知日志接口与手动重发
- 新增提醒规则配置页和通知日志页

## Non-Goals
- 不实现LINE/企业微信等其他通知渠道（V1明确只做邮件，PRD第二十一章
  21.1节说明这些是V2+的扩展方向）
- 不实现通知模板的可视化编辑器（V1的模板编辑是纯文本编辑
  subject_tpl/body_tpl字段，不做拖拽式的可视化邮件设计器）
- 不实现"已读回执"或"邮件是否被打开"这类追踪（超出了通知系统的
  核心职责，且涉及额外的隐私考虑）
- 不实现按学生/老师个体设置提醒偏好（提醒规则是系统级统一配置，
  不支持"张同学希望提前1小时收到提醒，李同学希望提前15分钟"这种
  颗粒度的定制）

## Impact
- Affected specs: notification（新增）、lesson-scheduling（MODIFIED，
  新增自动关闭过期课次这一状态转移路径）
- Affected code: backend/internal/notification/、backend/internal/job/、
  backend/internal/lesson/(自动关闭逻辑)、
  frontend/admin/src/views/notification/
- 依赖：005-add-lesson-scheduling（需要lesson.remind_sent_at字段）、
  007-add-payment-and-ledger（余额预警需要enrollment余额数据）、
  001-add-project-scaffold（需要system_config中的提醒规则配置项）
- 被依赖：无（本change是通知能力的完整闭环，后续change不直接依赖
  它的内部实现，只是008的工作台可能会展示"失败通知数"这类指标，
  但那是读取notification_log的独立查询，不构成强耦合）

## 风险与应对
邮件是外部依赖，Resend服务偶发不可用是正常情况，不属于本change的bug，
但系统必须能感知失败、记录失败、并在合理时间内自动重试，最终仍失败的
要能被人工看到（晨报的"失败通知"栏目）。
