## ADDED Requirements

### Requirement: 学生档案管理
系统必须支持学生档案的创建、查询、编辑和状态变更，且不因联系方式
缺失或重复而阻止真实业务的开展。

#### Scenario: 新建学生
- **WHEN** 提交合法的学生姓名(必填，1~50字符)及其他选填字段
- **THEN** 创建成功，status默认为ACTIVE

#### Scenario: 姓名长度校验
- **WHEN** 提交的姓名为空或超过50字符
- **THEN** 返回40001参数校验失败

#### Scenario: 邮箱重复不阻止创建
- **WHEN** 新建学生的邮箱与已有学生重复
- **THEN** 创建仍然成功，前端应提示"该邮箱已存在学生：XXX"供人工判断

#### Scenario: 无邮箱允许保存但需提示
- **WHEN** 新建学生不填写邮箱
- **THEN** 创建成功，前端应提示"未填写邮箱，将无法接收自动提醒"

#### Scenario: 状态变更留痕
- **WHEN** 学生状态从ACTIVE变更为PAUSED或ENDED
- **THEN** operation_log新增一条对应记录，记录操作人和变更前后状态

#### Scenario: 学生软删除不影响历史
- **WHEN** 某学生已产生过课次或充值记录，之后被标记deleted_at
- **THEN** 历史课次/充值记录仍能正常查询到该学生的姓名等基础信息

### Requirement: 家长联系方式管理
系统必须支持一个学生关联多条家长联系方式，并标记主联系人，且任意
时刻最多只有一条主联系人记录。

#### Scenario: 新增多条家长联系方式
- **WHEN** 给同一学生添加第二条家长联系方式
- **THEN** 创建成功，两条记录都能查询到

#### Scenario: 设置主联系人
- **WHEN** 对某条家长记录调用设为主联系人
- **THEN** 该学生下此前的主联系人标记被取消，新记录变为唯一主联系人

#### Scenario: 允许暂无主联系人
- **WHEN** 学生的所有家长联系方式都未被设为主联系人
- **THEN** 系统不强制要求必须有一条主联系人，此状态合法
