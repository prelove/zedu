# Design: 档案数据模型设计考虑

## 为什么student/teacher只做软删除
一旦学生或老师产生过课次记录、账务流水，物理删除会导致这些历史记录
出现"悬空引用"（关联的student_id/teacher_id在表中查不到对应记录），
破坏审计可信度。因此student.status/teacher.status只有ACTIVE/PAUSED/
ENDED三个业务状态，deleted_at字段仅用于极少数"完全录错、从未产生过
任何业务记录"的场景下的技术性清理，且这个清理动作不通过常规CRUD
接口暴露，只能通过后台脚本执行（本change不实现该清理脚本，只预留
字段）。

## parent为什么是1对多而非1对1
真实场景中一个学生可能有父亲、母亲两个联系人，甚至祖父母代为联系的
情况，is_primary字段标记"当前最优先联系谁"，而不是限制只能有一个
联系人记录。设为主联系人时，需要在同一事务内把该学生下此前的主
联系人标记清除，保证任意时刻最多只有一条主联系人记录（不要求
必须有一条，允许学生暂时没有设置主联系人）。

## teacher_capability的唯一性约束设计
约束是(teacher_id, track_id, level_id)三元组唯一，而不是包含domain_id
的四元组——因为track_id已经能唯一确定其所属的domain_id（track表有
外键指向domain），加入domain_id到唯一约束里是冗余的，反而可能在
数据不一致时产生误判。

能力记录允许level_id为NULL（表示"能教这个方向，但暂不区分具体等级"，
适用于某些还没有精细分级的方向或体育/艺术类课程初期），此时唯一性
约束按SQLite的NULL语义处理（多条track_id相同但level_id都为NULL的
记录不会被唯一约束拦截，因为SQL标准中NULL不等于NULL）——这是一个
需要在测试中明确覆盖的边界情况。

## 课程体系维护为什么用三层联动而非扁平列表
如果把course_domain/track/level做成三个独立的扁平管理页面，运营者
在新建一个方向时需要先跳到领域列表复制ID再跳回来，非常不直观。
三层联动（点领域看方向，点方向看等级）符合运营者的实际心智模型：
"我要给日语加一个新的学习路线"这个操作，天然是"先确定属于哪个领域"
再往下钻取，这是根据PRD第十章10.6节的UI规格倒推出的交互逻辑。

## source_channel字段的定位
student.source_channel是一个选填的自由文本字段，记录"这个学生是
通过谁/什么渠道介绍来的"，这不是一个强制结构化的枚举（因为介绍人
可能是任何人，做成外键关联另一张"渠道"表对V1来说过度设计），仅作为
运营者自己的备忘，不参与任何业务逻辑判断。
