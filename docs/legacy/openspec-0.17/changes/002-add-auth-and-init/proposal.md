# Proposal: 用户认证与首次初始化向导

## Why
Zedu系统需要基础的登录认证能力，以及首次部署时的课程模板初始化流程，
这是后续所有功能的前置依赖。同时这也是"一客一部署"策略（PRD第一章
1.6节）能否真正落地的关键——如果初始化向导做得足够通用，未来给做
K12、体育、艺术培训的朋友复制部署时，才不需要改代码，只需要在向导
里选另一个模板。

## 业务背景
PRD第二章"产品设计五大原则"中的原则一"结构上通用，配置上灵活"要求
系统不能在代码里硬编码"日语"这个学科。本change的初始化向导正是这个
原则第一次在用户可见的界面上体现——运营者第一次打开系统看到的不是
"日语教务系统"的固定标题，而是一个可以选择模板（或从空白开始）的
引导页，这个设计选择本身就是在传达"这套系统不是为日语量身定做的
单一用途工具"这个产品定位。

同时，安全性上的考虑也不能因为是"内部小系统"而放松：PRD第二十章
20.1节明确了密码存储、登录防暴力、Token管理三项安全要求，本change
是这些要求第一次被具体实现的地方。

## What Changes
- 新增 user_account 表的认证接口（登录/刷新/登出/改密码）
- 新增首次启动自动创建Owner账号的逻辑，密码只打印到日志不通过任何API暴露
- 新增初始化向导接口，检测course_domain是否为空并支持应用种子模板
  （日语模板 / K12模板 / 空白，对应001已准备好的种子数据）
- 新增登录页、初始化向导页面、强制改密码流程

## Non-Goals
- 不实现"记住我"/"多设备同时登录管理"这类增强登录体验的功能
- 不实现OAuth第三方登录（微信/Google登录等），V1只有用户名密码
- 不实现自定义模板上传（初始化向导只在日语/K12/空白三个预置选项中选，
  不支持运营者自己上传一份模板JSON）

## Impact
- Affected specs: auth（新增）、init（新增）
- Affected code: backend/internal/auth/、backend/internal/init/、
  frontend/admin/src/views/login/、frontend/admin/src/views/onboarding/
- 依赖：001-add-project-scaffold（需要user_account表和course_domain表
  已建好，且需要001准备好的日语/K12模板种子SQL文件可供本change调用）
