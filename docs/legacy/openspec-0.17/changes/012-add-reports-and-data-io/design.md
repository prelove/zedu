# Design: 报表聚合查询与导入幂等处理

## 图表数据聚合的查询设计原则

五类图表数据接口都应该遵循同一个原则：**在数据库层面用SQL的GROUP BY
和聚合函数完成统计，而不是把明细数据全部取到应用层再用Go代码做循环
统计**。这不仅是性能考虑（虽然V1数据量级下性能差异不明显），更是
为了避免统计口径不一致的风险——如果应用层的统计逻辑和数据库层的
筛选条件出现细微差异（比如漏加了一个status过滤条件），会产生难以
察觉的数字错误。

- 月度收入趋势：按月GROUP BY对lesson_finance.charge_amount和
  teacher_pay_amount求和，取最近12个月
- 学生增长曲线：按月GROUP BY student.created_at计数，累加得到
  累计活跃学生数
- 课程方向分布：按track_id GROUP BY对lesson_finance求和，计算占比
- 老师带课分布：按teacher_id GROUP BY对attendance.lesson_deducted求和
- 课时完成率：按月GROUP BY lesson.status分类计数(COMPLETED/CANCELLED/总计)

所有涉及百分比计算的图表（如课程方向占比），要注意浮点数精度和
四舍五入导致多个百分比相加不等于100%的情况，前端展示时可以接受
这种细微误差（如39.9%+30.1%+30.1%=100.1%），不需要额外做"调整
最后一项使总和精确为100%"这种过度设计。

## Excel导入的幂等与部分失败处理

导入接口的核心挑战是"一个Excel文件里有100行数据，其中3行有问题
（邮箧重复/必填字段缺失），如何处理"。设计方案：
1. 整个导入过程**不使用单一大事务**（不同于账务操作要求的强一致性），
   而是**逐行独立提交**——每一行数据的校验和插入是独立的，第N行
   失败不影响第N+1行的处理，这是导入场景的正确设计，因为运营者
   期望的行为是"尽可能多导入成功，把失败的行单独列出来改"，而不是
   "一行出错全部回滚重来"
2. 每一行的处理结果（成功/跳过/失败及原因）都记录到一份"导入报告"
   中，作为接口响应的一部分返回给前端，前端展示成一个可读的表格
3. 邮箧重复的判断：查询时使用大小写不敏感匹配（避免"a@b.com"和
   "A@B.com"被误判为不同邮箱导致重复导入）

## 导入模板的字段设计

学生导入模板固定列：姓名(必填)/邮箱/电话/学习方向(选填，需与已有
课程体系的方向名称匹配)/当前等级(选填)/备注。若"学习方向"列填写的
名称在系统课程体系中找不到匹配，该行仍然导入学生基础信息成功，
但不会自动创建课程报名（enrollment），需要运营者导入后手动补充，
导入报告中应明确标注"学生已创建，但方向XXX未匹配，请手动创建
课程报名"这类提示，而非静默忽略或报错整行失败。

## 报表导出的格式一致性

Excel导出使用excelize库，PDF导出使用gofpdf库，两者呈现的数据内容
必须完全一致（同一份财务报表，Excel版和PDF版看到的数字应该相同），
避免出现"因为用了两套不同的聚合查询代码导致两个格式的报表数字对
不上"这种情况——正确做法是两个导出接口共享同一个"生成报表数据"的
service层函数，只是最后一步用不同的库把同一份数据结构渲染成不同
的文件格式。
