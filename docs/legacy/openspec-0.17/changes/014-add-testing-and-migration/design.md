# Design: 端到端测试执行方式与迁移流程

## 为什么本change的specs是"过程性"而非"功能性"

前面13个change的specs都是"系统必须支持XX功能"这种功能性需求，
本change不同——它验证的是"已经实现的功能组合在一起是否可靠"，
因此这里的Requirement更接近"验收标准"而非"待实现的能力"。这是
OpenSpec框架用于"发布验收类change"的一种合理变体：Scenario描述
的是验证过程本身应该得出的结果，而非某个API的行为。

## 端到端测试与各change单元测试的关系

TC-01~TC-21中的每一条，在原本对应的change（如TC-01~07对应006）
里，已经以mock数据或内存数据库的单元测试/集成测试形式验证过。
本change的复测方式是：**在编译好的完整二进制上，用真实的SQLite
文件（非内存数据库），通过HTTP请求（而非直接调用service层函数）
触发这些场景**，这样能捕捉到"各层组装在一起后才会出现的问题"，
比如路由注册遗漏、中间件顺序错误、跨change的数据依赖关系是否
正确（如009的换老师批量更新是否真的与004的换老师逻辑正确衔接）。

## 真实数据迁移的流程设计

1. 运营者提供脱敏后的现有Excel（学生+老师两份）
2. 用012实现的导入接口执行正式导入，产出导入报告
3. 人工核对导入报告中的"跳过"和"未匹配"行，逐条与运营者确认
   如何处理（补充课程报名、修正邮箱冲突等）
4. 核对导入后的学生/老师总数、抽样核对5~10个学生的关键字段
   （姓名、余额如有预先设定）与原Excel完全一致
5. 进入并行运行期：这期间运营者仍然用Excel记录日常业务，同时
   Operator把同样的操作在Zedu里重复录入一遍，两周后核对两边的
   学生数、课次数、账务汇总数字是否一致

这个"双轨录入"的过程虽然对运营者和Operator是额外负担，但是建立
对新系统信任、发现"系统实现和真实业务理解有偏差"这类问题的必要
成本，不建议跳过或压缩这个观察期。

## 备份恢复演练的具体步骤

1. 在生产环境（或高度仿真的预发布环境）执行一次手动备份
2. 人为破坏当前数据库（如删除或损坏data/zedu.db文件，模拟真实
   故障场景）
3. 用备份文件恢复（按PRD第二十章设计的恢复流程，恢复前应先对
   "损坏后的当前状态"做一次快照备份，即使这个状态是损坏的，这是
   为了保留完整的故障现场供事后分析）
4. 验证恢复后的系统能正常启动，数据与备份时刻一致
5. 记录整个演练耗时，作为未来真实故障发生时的时间预期参考

## 24小时稳定性观察的具体检查点

不是单纯"让服务跑一天"，而是在24小时内验证：
- 六类定时任务（010）是否都按各自的Cron表达式实际触发过至少一次
  （课前提醒、晨报、余额预警是每日级别，通知重试是30分钟级别，
  这些在24小时内应该有多次触发记录可查）
- 服务进程本身没有因为未捕获的panic而意外退出（检查进程运行时长
  和日志中是否有崩溃堆栈）
- 内存占用没有出现持续增长的迹象（简单检查前后内存占用，不需要
  专业的内存泄漏分析工具，V1阶段这个层面的检查足够）
