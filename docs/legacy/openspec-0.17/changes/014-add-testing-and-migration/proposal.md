# Proposal: 全量测试、真实数据迁移与正式上线

## Why
001~013累计构建了完整的V1功能，但从未做过一次"把所有change拼在
一起，用真实数据完整跑一遍"的验证。这个change不新增任何业务功能，
而是项目从"一堆通过了各自单元测试的change"变成"一个真正可以交给
运营者使用的产品"的最后一道关卡。

## 业务背景
PRD第二十四章列出的TC-01~TC-21测试用例，覆盖了账务事务、配置化
边界、幂等性状态机、数据安全、跨平台部署五大类，这些用例中的很多
条目在各自change的tasks.md中已经以单元测试或集成测试的形式覆盖过
一次，但本change要求的是**端到端、使用真实环境和真实数据规模**的
复测——单元测试用mock数据验证逻辑正确性，本change要验证的是
"这些正确的逻辑组装在一起，在接近生产的环境下是否仍然正确"。

真实数据迁移是运营者能否真正切换到Zedu的临门一脚：PRD明确要求
"并行运行观察期"（建议2周），在这期间新旧系统的数据必须能核对一致，
这是建立运营者对新系统信任的关键环节，不能跳过或缩短。

## What Changes
- 按PRD第二十四章TC-01~TC-21逐项执行端到端测试，记录测试报告
- 用运营者提供的真实（脱敏）Excel数据执行一次正式的生产环境导入
- 建立新旧系统并行运行流程，记录核对结果
- 执行一次真实的"备份→模拟数据损坏→恢复"演练
- 24小时稳定性观察（服务持续运行，验证定时任务按预期触发）

## Non-Goals
- 不实现超出PRD第二十二章非功能性需求目标值的性能压测（不做
  "1万并发用户"这类超出V1实际规模的极限测试）
- 不实现渗透测试级别的安全审计（基础安全检查已在各change中通过
  测试覆盖，本change不额外引入专业安全审计工具或流程）
- 不在本change中处理"迁移历史课次/充值记录"（PRD附录C已明确本次
  部署只迁移当前余额快照，不迁移历史流水，历史流水从上线日起
  重新产生）

## Impact
- Affected specs: testing-and-migration（新增，以过程性验收标准
  为主，而非典型的功能性Requirement）
- Affected code: 本change原则上不产生新的业务代码，但测试过程中
  发现的Bug修复会散布在对应能力所在的模块中（如在
  backend/internal/lesson/中修复某个边界条件遗漏）
- 依赖：001~013全部change必须已完成并archive
- 被依赖：无（本change是V1发布前的最后一道关卡）
