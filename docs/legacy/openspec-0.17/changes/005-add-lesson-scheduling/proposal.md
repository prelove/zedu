# Proposal: 排课管理（基础版）

## Why
排课是连接课程报名和课后确认的中间环节，也是运营者日常操作频率最高
的功能之一。本change只做基础CRUD，时间冲突检测和取消功能留到009增强，
避免一次change范围过大导致review困难。

## 业务背景
根据PRD第九章9.4节的排课流程，一节课的创建需要综合选择"哪个学生的
哪个课程项目、由哪位老师、什么时间、多长时长、用什么方式上课"，
这个信息密度较高的表单要有合理的默认值联动（比如选定学生后自动列出
该学生的ACTIVE课程项目，选定项目后自动带入当前负责老师），减少
运营者的重复选择负担。

时长字段（duration_min）刻意不假设固定60分钟——PRD第八章8.1节明确
指出小型教培机构的实际课时可能是30/50/60/90分钟等各种情况，如果
系统只支持60分钟整课，会直接限制系统对其他业务类型（比如50分钟制
的日语会话课、90分钟的强化课）的适用性。

## What Changes
- 新增lesson基础CRUD（创建/查询/编辑，暂不含取消和冲突检测）
- 新增排课列表页和新建课次表单

## Non-Goals
- 不实现时间冲突检测（留给009）
- 不实现课次取消功能（留给009，因为取消涉及是否需要通知已排课的
  学生老师，与009的通知增强场景放在一起更合理；本change的编辑
  接口允许SCHEDULED状态下修改字段，间接可以用于"改期"场景）
- 不实现日历视图（本change只做列表视图，日历视图留给前端后续
  迭代，不阻塞后端核心逻辑）

## Impact
- Affected specs: lesson-scheduling（新增）
- Affected code: backend/internal/lesson/、frontend/admin/src/views/lesson/
- 依赖：004-add-enrollment-assignment（需要enrollment和assignment已存在）
- 被依赖：006（课后确认需要lesson记录）、009（冲突检测和取消增强）、
  010（提醒任务需要lesson.remind_sent_at字段）
