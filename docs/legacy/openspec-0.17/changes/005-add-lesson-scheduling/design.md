# Design: 课次创建的字段联动与编号生成

## lesson_no生成策略
lesson_no是面向人类可读的业务编号（区别于自增的数据库id），格式建议
`L` + `YYYYMMDD`（创建日期）+ 4位当日序号，如`L202607060001`。生成
逻辑需要处理"同一天内序号递增且不重复"，最简单可靠的实现是在事务内
先查询当天已有的最大序号再+1，而不是维护一个全局计数器（避免多实例
部署时的计数器同步问题，虽然V1是单实例，但这个设计选择让未来扩展
更从容）。

## 学生→课程报名→老师的表单联动逻辑
新建课次表单选定学生后，前端应调用`GET /students/{id}/enrollments`
（003/004已实现）并只展示status=ACTIVE的项目；选定项目后，前端应
查询该enrollment当前role_type=MAIN且status=ACTIVE的assignment，
自动带入对应老师作为默认值，但保留可改选的下拉（应对代课等场景）。
这个联动逻辑主要是前端职责，后端只需保证enrollment和assignment的
查询接口能被正确调用。

## 时区处理的具体约定
`scheduled_start_at`/`scheduled_end_at`两个字段在数据库中以UTC存储，
`timezone`字段记录创建时使用的显示时区（默认Asia/Tokyo），前端展示
时用这个timezone字段做本地化转换。这个设计允许未来学生/老师分布在
不同时区时，仍能各自看到符合自己习惯的时间显示（V1暂不实现"按
收件人时区分别显示"这个高级特性，但数据结构已经支持，为210
notification change的双时区显示留出扩展空间）。

## 上课方式与链接的宽松校验
`meeting_link`字段设计为选填而非必填，因为"微信群"这种上课方式
可能没有一个正式的URL链接，只是一个群名/群号的说明，这个字段的
校验应该宽松（允许任意文本，不强制URL格式），把"链接是否有效"
这个判断留给运营者自己负责，系统不做过度校验。
