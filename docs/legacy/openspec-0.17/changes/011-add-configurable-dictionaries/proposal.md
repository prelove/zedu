# Proposal: 配置化字典管理与付款凭证

## Why
这是"默认值可以给，业务不能写死"这条核心设计原则（PRD第二章原则二）
在管理界面上的具体落地。此前的change（001种子数据、006出勤分类读取、
007支付方式读取）都只做了"读"，本change补上"写"——让运营者能在不改
代码的情况下增删支付方式、调整出勤分类的建议比例。

付款凭证则解决小机构现金/转账收款场景下"到底有没有付、付了多少"的
真实纠纷隐患（PRD第七章7.4节）。

## 业务背景
PRD第一章1.6节明确了"一客一部署"的复制策略——未来可能把Zedu部署给
做K12、体育、艺术培训的其他朋友使用。这些机构的支付习惯、出勤处理
规则很可能与日语机构不同（比如某机构可能常用支付宝而非PayPay，
某体育培训机构的"请假"处理逻辑可能与语言培训机构完全不同）。如果
这些字典是硬编码的，每次给新机构部署都要改代码重新编译，这与"轻量
可复制"的产品定位直接矛盾。本change让这些差异变成纯粹的后台配置项，
是"一套代码，服务多个不同业务"这个愿景能否成立的关键一环。

付款凭证的设计动机来自更朴素的现实：小机构的收款方式往往是家长
直接微信转账或发红包，运营者收到后需要留存截图作为"确实收到了这笔
钱"的证据。如果系统里的充值记录只是一串数字而没有原始凭证支撑，
一旦学生或家长事后对充值金额有异议，运营者将很难自证清白。

## What Changes
- 新增支付方式的创建/编辑接口（此前只有007的只读接口）
- 新增出勤分类的创建/编辑接口，含建议值三列的编辑（此前只有006的只读接口）
- 新增payment_attachment表及上传/查询/删除接口
- 新增充值表单的凭证上传按钮、支付方式管理页、出勤分类管理页

## Non-Goals
- 不实现字典项的物理删除（只支持enabled=0禁用，理由与student/teacher
  的软删除原则一致：一旦被历史记录引用，删除会导致悬空引用）
- 不实现凭证文件的图片编辑功能（裁剪、压缩、旋转等，V1只做"上传原图、
  查看原图"这个最基础的能力）
- 不实现凭证的OCR自动识别金额（不做"拍照自动填充充值金额"这类智能化
  功能，凭证纯粹是佐证材料，金额仍由Operator手动录入）
- 不实现字典项的导入导出（如批量导入100个支付方式，V1阶段字典项
  数量很小，一个个手动维护即可）

## Impact
- Affected specs: configurable-dictionaries（新增，覆盖支付方式管理、
  出勤分类管理、付款凭证三部分）
- Affected code: backend/internal/system/(字典写接口)、
  backend/internal/finance/(payment_attachment)、
  frontend/admin/src/views/system/dictionaries/、
  frontend/admin/src/views/finance/payments/(凭证上传)
- 依赖：007-add-payment-and-ledger（需要student_payment表和
  payment_method只读接口）、006-add-attendance-confirmation（需要
  attendance_outcome_type只读接口）
- 被依赖：无（本change是配置化能力的收尾，不阻塞任何其他change的
  核心链路，是体验增强性质）

## 不在本change范围内
本位币的锁定逻辑（PUT /system/base-currency）已在006完整实现，
本change只新增系统设置页面里展示该状态的UI，不新增后端行为。
