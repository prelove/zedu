# Design: 字典写操作与凭证文件管理

## 字典code不可变原则

payment_method.code和attendance_outcome_type.code一旦被业务表引用
（student_payment.payment_method_code 或 attendance.outcome_type），
就不应再允许修改，否则会破坏历史记录的语义。方案：
- 编辑接口只允许修改name（展示名称）和sort_order/enabled，
  不接受对code字段的修改请求（即使传了也忽略，不报错，因为这是
  一个"防呆"设计而非用户输入错误——前端表单设计上也应该将code
  字段在编辑模式下设为只读展示，从UI层面就避免用户产生"这个可以改"
  的错误预期）
- 新增接口的code字段做唯一性校验，重复返回40901
- 禁用（enabled=0）而非删除：被禁用的字典项不再出现在新建充值/
  新建课后确认的选择列表中，但历史记录中引用了该code的数据仍能
  正常显示其name（因为name字段本身没有被删除，只是enabled状态变化）

## 出勤分类建议值编辑的空值语义

attendance_outcome_type的三个建议值列（suggested_deduct_lessons/
suggested_charge_ratio/suggested_teacher_pay_ratio）允许为NULL
（表示"无建议，前端不自动填充"）。编辑接口需支持显式传null来清空
某个建议值，不能用"不传该字段=不修改"这种语义（否则无法把一个
已有建议值的分类改成"无建议"）。约定：请求体中显式包含该字段且值为
null时，才更新为NULL；不包含该字段时，保持原值不变。

这个设计上的微妙之处在于：标准的Go struct JSON反序列化无法区分
"字段值为null"和"字段完全不存在于JSON中"这两种情况（两者都会
让对应的Go字段保持零值）。实现上需要使用`*float64`指针类型接收，
或者先将请求体解析为`map[string]interface{}`再逐个判断key是否存在，
才能正确实现这个语义区分。这是一个容易被AI工具用简单粗暴的方式
实现错的细节，需要在代码review时特别关注。

## 付款凭证文件管理

- 存储路径：`data/uploads/payments/{payment_id}/{uuid}_{original_filename}`，
  文件名加uuid前缀避免同名覆盖
- 上传时校验：文件类型（jpg/jpeg/png/webp/pdf的MIME和扩展名双重校验，
  防止伪造扩展名绕过——即检查文件的实际二进制内容(magic number)是否
  符合声称的类型，而不仅仅信任HTTP请求头里的Content-Type或文件名
  后缀）、文件大小(≤5MB)、同一payment下已有附件数(<3)
- 访问控制：文件下载接口必须校验JWT，不提供无鉴权的静态文件直链；
  即使知道文件路径也无法绕过Gin的鉴权中间件直接访问
- 删除：只允许删除未被作废(status=CONFIRMED)的充值下的附件，
  已作废的充值其附件保留不可删（保持作废记录的完整取证链）
- 上传失败的原子性：如果文件已经写入磁盘但数据库insert失败（或反之），
  不应留下孤儿文件或孤儿数据库记录——实现上应该先完成文件写入磁盘，
  确认写入成功后再插入数据库记录；若数据库插入失败，需要有清理逻辑
  删除已写入的孤儿文件，避免磁盘空间被无主文件占用且无法通过正常
  业务流程发现和清理

## 为什么附件数量上限(3)不做成可配置项

这看起来似乎违背了"默认值可以给，业务不能写死"的原则，但这里是一个
刻意的例外：附件数量上限本质上是一个技术性的防滥用保护（防止某个
充值记录被无限上传文件占满磁盘），而不是业务规则。业务规则（如
支付方式有哪些、出勤分类怎么分）才需要配置化，纯技术性的资源保护
阈值可以合理地保持代码常量，除非未来确实出现"3张不够用"的真实
反馈，再评估是否要开放这个配置项。
