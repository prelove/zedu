# Zedu 项目交付深访摘要

## 结论

本项目采用“整体规划、分阶段验收、持续追踪”的方式推进。路线图覆盖 MVP、V1、V1.5 与 V2；当前执行焦点为 MVP，MVP 通过验收后继续 V1。

## 关键问答

1. 事实源：确认 `docs/2_prd/Zedu-PRD-Final-v3.1.md` 为正式定案版和唯一业务事实源。
2. 阶段边界：先完成并验收 MVP，再进入 V1；不得丢失后续任务、依赖、状态和验收定位。
3. 决策权限：技术与文档治理可自主推进；产品范围、财务语义、权限、隐私、持续费用、生产部署和真实数据迁移需要人工确认。
4. 压力复核：原 PRD 的 MVP 与完整旅程验收口径冲突。最终决定将 Resend 邮件通知和付款凭证上传纳入 MVP；正式老师结款保留到 V1。

## 被修正的假设

- “58项澄清全部阻塞开发”不成立：v3.0 已通过配置化方案吸收多数问题，仅附录 C 的上线配置仍需确认。
- “通知与上传均在V1”不成立：它们是首个真实业务实例的常用能力，应进入 MVP。
- “上传只是简单文件接口”需要加安全边界：格式/大小校验、鉴权访问、孤儿清理和备份包含均为 MVP 验收条件。

## 剩余非阻塞事项

- PRD 附录 C 的实际运营配置可以用默认值开发，但必须在 MVP 上线验收前逐项确认。
- 正式生产部署、真实数据迁移和持续费用服务在相应阶段单独审批。
