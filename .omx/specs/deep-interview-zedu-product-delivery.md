# Zedu 项目交付执行规格

## 元数据

- 模式：standard deep interview
- 轮次：4
- 最终歧义度：0.12
- 门槛：0.20
- 项目类型：已有完整规格、尚无代码的绿地项目
- 上下文：`.omx/context/zedu-product-delivery-20260711T025757Z.md`
- 唯一业务事实源：`docs/2_prd/Zedu-PRD-Final-v3.1.md`

## 意图与结果

建立一个可由人类及 GLM、Devin、Kimi、Codex 等 AI 编码工具共同执行的 Zedu 交付体系：先验证和修订 Claude 产出的 PRD/OpenSpec，再建立规范、路线图、工程、测试与验收证据，交付 MVP 后继续推进完整 V1。

## 范围

### 当前执行范围：MVP

1. 登录与 Owner/Operator 基础权限、首次初始化。
2. 学生、老师、课程四层配置、报名和师生安排。
3. 充值、排课、课后确认、学生流水和老师应付查询。
4. 极简工作台和本地完整备份。
5. Resend 邮件通知：至少覆盖课前提醒及可追踪的发送日志、失败处理与幂等。
6. 付款凭证上传：数量/大小/真实文件类型校验、鉴权访问、孤儿文件清理、随备份保存。
7. 账务事务原子性、状态机、幂等、Windows 10 日文环境和 UTF-8 兼容验收。

### 后续范围

- V1：正式老师结款、完整通知任务、报表、导入导出、移动页面、打包和全量上线验收。
- V1.5/V2：按 PRD 触发条件追踪，不提前实现。

### 非目标

- MVP 不实现正式老师结款提交与结算流水。
- 不实现多租户 SaaS、学生/老师/家长登录、自动排课、复杂定价/课酬/请假规则、在线支付、原生移动 App、小程序、税务发票、题库作业。
- 不用空壳、TODO 或无法复现的声明冒充已交付。

## 决策边界

代理可自主决定：文档结构、OpenSpec 格式迁移、任务拆分和依赖、代码组织、测试方法、CI、编码兼容性，以及不改变业务语义的技术方案。

必须人工确认：产品范围增减、财务规则/金额语义、角色权限、个人信息处理、产生持续费用的云服务、正式生产部署和真实数据迁移。

附录 C 的运营配置允许使用 PRD 默认值开发，但必须在 MVP 上线验收前逐项确认。

## 约束

- 文档中文优先；产品支持中/日/英国际化。
- Windows 10 日文环境开发，仓库文本统一 UTF-8，脚本不得依赖本地代码页。
- OpenSpec 与 Superpowers 使用官方最新稳定版和对应工具的官方安装方式。
- 每个需求必须可追溯到 PRD、OpenSpec scenario、task、测试和验收证据。
- 财务多表写入必须单事务；金额禁止 float；历史财务事实不可被未来配置修改。

## 可测试验收条件

1. OpenSpec 所有当前 change 通过最新 CLI 严格校验。
2. MVP 范围形成明确的 proposal/design/spec/tasks、依赖图和验收矩阵。
3. 主链路可通过 UI/API 完整执行并人工核账一致。
4. 课后确认故障注入后全部回滚；重复提交不重复扣费或增加应付。
5. 邮件提醒具备幂等门控、发送日志和失败可追踪性；使用已配置的 Resend 凭证做受控集成验收，密钥不得进入仓库或日志。
6. 付款凭证不可匿名访问，伪造扩展名被拒绝，数据库失败不遗留孤儿文件，备份包含上传文件。
7. Windows 10 日文环境下中文/日文文件名、日志、CSV/Excel 和 UI 无乱码。
8. lint、类型检查、单元/集成/E2E 测试及静态分析通过，结果可复现。
9. README、路线图、状态表和验收报告与实际进度一致。

## 压力复核结果

复核了“通知、付款凭证是否应推迟到V1”的早期假设。用户基于真实运营频率和现有 Resend 可用性，将两者提升到 MVP；正式结款因当前业务不规范且非强约束，继续保留到 V1。由此 MVP 验收必须单独定义，不能直接照搬 PRD 24.1 的完整 V1 旅程。
