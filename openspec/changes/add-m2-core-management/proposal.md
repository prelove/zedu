## Why

M1 已交付可验证的工程基础，但系统尚不能让 Owner/Operator 登录、完成首次初始化，或维护学生、老师与课程报名等核心经营资料。M2 将这些高频、低财务风险的能力形成最小闭环，为后续充值、凭证、排课与通知提供可信的身份和主数据基础。

## What Changes

- 新增 Owner/Operator 认证、会话续期、退出和当前用户查询；学生、老师、家长均不创建账号、不允许登录。
- 新增首次启动初始化向导与可重复执行的基础模板初始化，只在尚未存在业务资料时允许重置模板。
- 新增学生、家长、老师、老师能力与可授时间的受控 CRUD；所有写入经 RBAC、审计和状态约束。
- 新增课程领域、方向、等级、能力标签字典，以及学生课程报名和师生安排的最小主数据链路；不创建课次、账务、通知或结款入口。
- **BREAKING（相对 PRD v3.1 第 9.1 节旧分支）**：学生邮箱为选填字段；一旦填写，必须全局唯一。创建或更新时与已有学生重复则拒绝写入并返回 `40901`；无邮箱仍可保存。
- 固化老师能力唯一性为 `(teacher_id, track_id, level_id)`，与 PRD R10 一致。

## Capabilities

### New Capabilities

- `operator-authentication`: Owner/Operator 登录、刷新、登出、当前会话和账号状态控制。
- `onboarding-initialization`: 首次启动初始化向导与受限、幂等的模板应用。
- `people-directory`: 学生、家长、老师、能力和可授时间的资料维护与审计。
- `course-enrollment-assignment`: 课程字典、课程报名和师生安排的主数据关系。

### Modified Capabilities

- 无；现有 M1 工程基础能力不改变其需求语义。

## Impact

- 后端：新增领域迁移、认证与 RBAC 中间件、密码哈希/令牌生命周期、审计记录及 HTTP 契约测试。
- 前端：新增受保护的登录、初始化和核心资料页面；保持 `zh-CN`、`ja-JP`、`en-US` key 对齐。
- 数据：新增用户、主数据和关系表；`student.email` 使用“非空时全局唯一（含软删除记录）”的约束，`teacher.email` 的约束策略在 design 中明确。
- 依赖与边界：不引入财务、付款凭证、排课、邮件发送、备份恢复、报表或正式老师结款的可执行 API、路由或 UI。

## Non-Goals

- 不实现学生/老师/家长登录、多租户、自动排课、课次、充值、流水、付款凭证、通知、备份恢复、报表或任何正式结款功能。
- 不提供“仍然新建”的绕过路径；重复邮箱不是 warning，而是阻断性数据冲突。

## Dependencies and Risks

- 依赖 M1 已验收的 SQLite 迁移、三语前端、CI 与日志基础。
- 依赖 PRD v3.1 第 3.2、4.1、9.1–9.3、13.2–13.3、14.2（R10）、16.3、20.1 和 24 章。
- 风险：认证与令牌边界若未先在 design 固化，容易形成不兼容 API；所有路径、令牌载荷、分页和响应格式必须在 design 后才冻结。
- 风险：邮箱唯一规则必须在前端预检、后端数据库约束、更新路径和软删除语义间保持一致，避免并发写入绕过检查。
