## Why

M2 已建立可审计的 Owner/Operator、学生、报名项目和课程主数据，但运营人员仍无法确认线下收款、保存付款凭证或核对学生余额。本 change 以最小财务闭环实现“人工确认充值 → 不可变流水 → 可鉴权凭证”的 MVP 主链路，并为后续排课、课后确认和备份提供可信的账务事实。

## What Changes

- 新增系统本位币读取与受限设置，以及支付方式字典的维护与初始化默认值。本位币只可在尚无财务事实时设置；首次成功充值在同一事务内锁定它。
- 新增 Owner/Operator 可创建、查询详情、分页筛选和作废的学生充值。充值保留原始金额、原币种、汇率快照、折算本位币金额、课时、付款方式、付款时间和备注；已确认记录不能编辑或删除。
- 新增学生账户流水查询。确认充值写入 `RECHARGE` 流水并更新报名余额；作废写入反向 `VOID` 流水并回滚余额，所有相关写入、审计和本位币锁定必须在单一事务中完成。
- 新增每笔充值最多三个付款凭证的上传、列表与经登录鉴权的下载。文件仅接受 `jpg/jpeg/png/webp/pdf`，单文件不超过 5 MiB，使用受控存储路径且不暴露匿名直链。
- 新增与上述后端契约一致的三语前端：学生账务 Tab、充值表单、充值记录/作废、凭证上传与查看入口，以及 Owner 系统设置中的本位币/支付方式最小维护界面。

## Capabilities

### New Capabilities

- `financial-configuration`: 本位币锁定规则和支付方式字典的受控维护。
- `student-recharge-ledger`: 学生充值确认、作废、余额更新与可追溯流水。
- `payment-evidence`: 付款凭证上传、列举和鉴权下载。

### Modified Capabilities

- 无；M2 已验收的认证、初始化、人员和报名能力不改变既有 Requirement。

## Impact

- 后端：新增 M3 migration、finance/payment 应用层、事务与审计封装、上传文件存储边界、稳定错误映射及 API 测试。
- 前端：新增 finance adapter、充值/流水/凭证和最小设置页面，保持 `zh-CN`、`ja-JP`、`en-US` key parity；沿用既定 HTTP 信封、认证和组件规模约束。
- 数据：在 005 migration 新建 `payment_method`、`student_payment`、`payment_attachment`、`student_account_ledger`，并补 `system_settings` 的 M3 默认配置。M2 003 实际只提供了设置与主数据表，不能假设 PRD DDL 已落库。
- 依赖：依赖 M2 已验收的认证、学生、报名和 operation_log；不新增第三方依赖，上传实现使用 Go 标准库和现有 SQLite 驱动。

## Non-Goals

- 不实现课次、出勤确认、老师流水、正式老师结款、退款/人工调整、财务报表、Excel/PDF 导出、支付网关、实时汇率、通知、备份恢复或公网文件分享。
- 不提供充值的编辑、删除或静默覆盖；纠错仅允许有原因的作废并保留冲正事实。
- 不实现图片裁剪、压缩、OCR、多级相册或超过三份凭证的文件管理。

## Dependencies and Risks

- 依赖 PRD v3.1 第 7.1–7.4、9.6、10.3/10.8/10.11、12.4、13.9/13.12、14.1–14.3（R8、R16、R18、R20）、15.3、16.3–16.5、18.1、20.1/20.3、23.2、24.3–24.6。
- 风险：金额精度、余额快照和作废冲正必须严格保持同一事务；任何失败不得留下 payment、ledger、余额或锁定标志的半完成状态。
- 风险：文件系统不具备数据库事务语义。design 将规定先落入临时受控位置、数据库成功后原子发布，并以补偿清理与恢复扫描处理失败，且不会把文件成功伪装为充值成功。
- 风险：M2 前端当前仍有延后执行的 Windows 全量 Vitest 门禁；该风险登记在项目状态中，不得把它误记为 M2 或 M3 的验收通过。
